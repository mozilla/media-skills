## Suggestions for Improvement

### Improving Project Tracking Over Time

1. **Automate the score cache on a schedule** — the script at `resources/scripts/wpt_webrtc_scores.py` works but requires SSL fixes for Python urllib on macOS. Consider switching to `requests` with certifi, or wrapping with `curl`. Running weekly against a cron would give automatic trend data without manual report generation.

2. **Add "first seen passing" timestamps** — the score cache JSON now captures per-test pass/total, but a separate table tracking when each test first reached 100% would let you calculate individual test lead times for retrospectives.

3. **Track multi-browser regressions separately** — `RTCRtpParameters-codecs`, `mandatory-getStats`, and `RTCRtpParameters-encodings` fail in multiple browsers. These are not in the FF-only burndown but may represent upstream WPT fixes that could benefit Firefox too. Consider a second tracking column: "Would pass if WPT fixed."

4. **Add a `?interop-2026` coverage check** — several tests exist in both canonical and `?interop-2026` variants. Automate a check to ensure the tracked variants are the right ones (some `?rest` variants have different subtest counts).

5. **Track WPT test issues separately** — 5+ tests are blocked on WPT fixes rather than Firefox bugs. A distinct "blocked-on-wpt" label in Bugzilla (or a spreadsheet column) would make it clearer what's actionable vs. what needs upstream work.

### Ways the Team Can Improve Bugzilla Data

1. **File bugs for untracked FF-only failures** — `RTCRtpSender-getStats.https.html` (1 fail) and `protocol/transceiver-mline-recycling.html` (2 fails) have no bugs. They are small but visible on the dashboard.

2. **Add WPT test paths to bug summaries or whiteboard** — Currently it requires reading notes to determine which WPT test a bug covers. A `[wpt: path/to/test.html]` convention in the whiteboard would make automated bug↔test mapping feasible.

3. **Close or WONTFIX bugs for confirmed WPT test issues** — Bugs like [2019361](https://bugzilla.mozilla.org/show_bug.cgi?id=2019361) (RTCSctpTransport-events correctness) have been open for months. If the conclusion is "test is wrong," WONTFIXing the Bugzilla entry and filing a WPT upstream issue clarifies the scorecard.

4. **Set priority fields on the child bugs** — Most of the 42 child bugs have no `priority` set. Even a simple P1/P2/P3 pass by Byron would help surface what's actively being pursued vs. parked.

5. **Link RTCTransportStats child bugs to each other** — The 6 stats sub-bugs (2019331, 2019333, 2019349, 2019355, 2019356, 2019389) are all independent items under 1225723 but don't cross-reference each other. A team member picking up one should know the others exist so they can batch the work.
