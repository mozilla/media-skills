## Suggestions for Improvement

_(Unchanged from March 27 report — no new issues identified.)_

### Improving Project Tracking Over Time

1. **Automate the score cache on a schedule** — `resources/scripts/wpt_webrtc_scores.py` works but requires SSL workaround on macOS. Switch to `requests` + `certifi`, or use `curl -k` wrapper, and run weekly for automatic trend data.

2. **Add "first seen passing" timestamps** — the score cache now captures per-test pass/total. A separate table tracking when each test first reached 100% would enable per-test lead-time calculation for retrospectives.

3. **Track multi-browser regressions separately** — `RTCRtpParameters-codecs`, `mandatory-getStats`, and `RTCRtpParameters-encodings` fail in multiple browsers. Not in the FF-only burndown but worth watching: upstream WPT fixes could benefit Firefox too. Consider a "blocked-on-wpt" or "multi-browser" column.

4. **Track WPT test issues separately** — 5+ tests are blocked on WPT fixes. A `[blocked-on-wpt]` whiteboard tag in Bugzilla would make what's actionable vs. upstream-dependent more visible.

### Ways the Team Can Improve Bugzilla Data

1. **Add WPT test paths to bug whiteboards** — A `[wpt: webrtc/path/to/test.html]` convention would enable automated bug↔test mapping.

2. **Set priority fields on child bugs** — Most child bugs have no `priority`. Even a P1/P2/P3 pass by Byron would surface what's actively pursued vs. parked.

3. **Link RTCTransportStats child bugs to each other** — The 6 stats sub-bugs (2019331, 2019333, 2019349, 2019355, 2019356, 2019389) don't cross-reference each other. Whoever picks one should batch them.

4. **Close or WONTFIX bugs for confirmed WPT issues** — Bugs like [2019353](https://bugzilla.mozilla.org/show_bug.cgi?id=2019353) (setLocalDescription-answer wrong error type) are Firefox-side tracking bugs for WPT fixes. WONTFIXing the Firefox bug and linking the WPT upstream issue clarifies the scorecard.
